TrueSkill via MCMC — A hands‑on walkthrough

This post walks through building a TrueSkill model and performing MCMC inference end‑to‑end: from matches → posterior skills → rankings → win‑probability queries. It’s a practical guide distilled from Assignment 2 (Q2).

1) Problem setup

We observe a set of matches. Each match is between Team A and Team B (one or more players per team) with a binary outcome (A wins vs B wins). Our goal is to infer latent skill values for each player and answer queries such as Which team is stronger? or What’s the probability A beats B?

2) Generative model (TrueSkill‑style)

  • Prior over skills: for each player i, s_i ∼ Normal(μ0, σ0²).
  • Performance: when a match occurs, player i’s performance is p_i = s_i + ε_i with ε_i ∼ Normal(0, β²).
  • Team performance: sums of member performances, e.g. P_A = Σ_{i∈A} p_i, P_B = Σ_{j∈B} p_j.
  • Outcome: A wins if D = P_A − P_B + ξ > 0 where ξ is match noise (probit/logit link; probit keeps everything Gaussian).

This yields a posterior p(s | matches) proportional to the product of win likelihoods and Gaussian priors. Closed forms are messy, so we use MCMC.

3) Data to factor graph

We construct a sparse design matrix that maps players to teams and matches. For each match m, compute a signed sum of involved player skills to form the difference score Δ_m = Σ s_i − Σ s_j. The likelihood uses Φ(Δ_m / σ) (probit) or σ(Δ_m) (logit).

4) Log posterior and (optional) gradient


# log posterior up to a constant (probit)
logp(s) = Σ_m log Φ(y_m * Δ_m / σ)  +  Σ_i log N(s_i; μ0, σ0²)

# If using a logistic link, replace Φ with sigmoid σ(x) = 1/(1+e^{-x}).
# Gradients can be derived analytically or computed via autodiff.

5) Samplers

  • Random‑walk Metropolis: simple, but tuning‑heavy and mixes slowly in correlated posteriors.
  • Preconditioned Metropolis: scale proposals by player frequency to improve acceptance.
  • Hamiltonian Monte Carlo (HMC): uses gradients to explore along posterior contours; higher effective sample sizes per step.

HMC step (conceptual)


# s: current skills (vector), U(s) = -logp(s)
# L leapfrog steps of size ε
v ~ Normal(0, I)
(s', v') = Leapfrog(s, v, ε, L, ∇U)
accept with prob min(1, exp(H(s,v) - H(s',v')))

Start with small ε and moderate L. Target an acceptance rate around 0.6–0.8; adapt ε during warm‑up.

6) From samples to insights

  • Player/Team rankings: rank by posterior mean skill; report uncertainty with credible intervals.
  • Win probabilities: to estimate P(A beats B), draw s from the chain and compute Δ = Σ_{i∈A}s_i − Σ_{j∈B}s_j; with probit, P = Φ(Δ/σ). Average over samples.

# Monte Carlo query
probs = []
for s in samples:
    delta = sum(s[i] for i in teamA) - sum(s[j] for j in teamB)
    probs.append(Phi(delta / sigma))   # or sigmoid(delta) for logit
P_A_beats_B = mean(probs)

7) Diagnostics

  • Trace plots: look for stationarity and mixing across players.
  • Autocorrelation / ESS: thin only if needed; prefer better tuning.
  • R‑hat: run multiple chains and check convergence (≈1.0).

8) Practical tips

  • Center skills at the prior mean to improve numerical stability.
  • Use a smaller β (performance noise) when matches are decisive; larger when volatile.
  • Precompute Δ_m with sparse ops; gradients become cheap matrix‑vector products.
